# skylight-login — closing the OAuth loop

**Date:** 2026-06-17
**Status:** Approved (design)

## Problem

The real Skylight login is OAuth 2.0 **Authorization Code grant with PKCE** (RFC 7636),
with a fixed `redirect_uri = skylight-family://welcome` — a custom URL scheme, not an
`http://127.0.0.1` loopback. Because the redirect target is a custom scheme, the standard
CLI "spin up a localhost HTTP server and use it as the redirect URI" pattern does not work:
the authorization server will only redirect to the registered custom scheme.

A `skylight-family://` scheme handler is already registered on the user's machine
(`~/.local/share/applications/skylight-family-handler.desktop` →
`~/.local/bin/skylight-family-handler`), but it currently only copies the callback URL and
shows a desktop notification. The token never reaches the CLI.

We want to close the loop so a single command authenticates and leaves the CLI ready to
make API calls.

## Constraints

- **`cli/` is fully generated** by `onlycli` and marked `// Code generated by onlycli; DO NOT EDIT.`
  The generated `cli/runtime/auth.go` implements device-code / client-credentials flows that
  do **not** match the real service. Login logic must **not** live in generated code.
- The generated CLI reads its bearer token from `~/.config/skylight/config.json` (mode 0600),
  per-profile `token` field. That file is the integration point: whatever we build must end up
  writing `access_token` there.
- The generated `Config`/`Profile` schema has **no** `refresh_token` field; we will not edit
  generated code to add one.
- Target platform is Linux desktop (XDG, `xdg-open`, `.desktop` scheme handlers).

## Approach

A standalone Go binary `skylight-login`, kept entirely separate from `cli/` so
`onlycli generate` never touches it. It uses a Unix-domain socket for IPC: the `login` mode
listens, the OS-launched scheme handler (`callback` mode) relays the callback URL back over
the socket. This is the user's proposed "open a socket, handler sends the token back" design,
and it is the only clean option given the custom-scheme redirect.

### Placement

- New directory `login/` at repo root with its own `go.mod`; binary name `skylight-login`.
- No file under `login/` carries a generated-code marker; regeneration of `cli/` cannot affect it.

### Two modes (subcommands of one binary)

#### `skylight-login login [--profile NAME]`

1. Generate PKCE material: `code_verifier` (43+ chars, RFC 7636 unreserved charset),
   `code_challenge = base64url(SHA-256(code_verifier))` with `code_challenge_method=S256`,
   and a random opaque `state` (CSRF protection).
2. Listen on a Unix socket at `$XDG_RUNTIME_DIR/skylight-login.sock`. Remove any stale socket
   file first. If `XDG_RUNTIME_DIR` is unset, fall back to a path under the config dir. The
   runtime dir is user-only (0700), acceptable for a secret in transit.
3. `xdg-open` the browser at:
   ```
   {base_url}/oauth/authorize
     ?response_type=code
     &client_id=skylight-mobile
     &redirect_uri=skylight-family://welcome
     &scope=everything
     &state={state}
     &code_challenge={challenge}
     &code_challenge_method=S256
     &prompt=login
   ```
4. `Accept()` one connection with a ~3-minute timeout; read the callback URL bytes.
5. Parse the callback URL. If it carries `error=…`, surface it and exit non-zero. Verify
   `state` matches the value generated in step 1. Extract `code`.
6. `POST {base_url}/oauth/token` (form-encoded) with:
   `grant_type=authorization_code`, `client_id=skylight-mobile`, `code`, `code_verifier`,
   `redirect_uri=skylight-family://welcome`.
7. On success, write `access_token` into the chosen profile's `token` in
   `~/.config/skylight/config.json`, preserving mode 0600. Save the rotating `refresh_token`
   to a sidecar file `~/.config/skylight/{profile}.refresh` (mode 0600) so a future refresh
   command can consume it without editing the generated config schema.
8. Remove the socket; print a success message.

#### `skylight-login callback <url>`

Invoked by the OS via the `.desktop` handler (`Exec=skylight-login callback %u`).

1. Try to connect to the socket. If a login is in flight (listener present), write the raw
   callback URL and exit.
2. If the connection fails (no login running), fall back to the current handler behavior:
   `wl-copy` the URL and `notify-send` — so a stray scheme click still does something sensible.

### Config / base-URL resolution

Reuse the CLI's resolution order: selected profile → `SKYLIGHT_BASE_URL` / `SKYLIGHT_TOKEN`
env → default `https://app.ourskylight.com`. The tool reads and writes the same `config.json`,
targeting the active profile (or `--profile`).

### `.desktop` change

`Exec` becomes `skylight-login callback %u`, replacing the bash handler. The existing bash
handler may be retired or kept as a manual backup.

## Error handling

- Stale socket file removed on `login` startup.
- `state` mismatch → reject and exit non-zero.
- `error=…` in the callback → surface the provider error and exit non-zero.
- Token-exchange non-2xx → surface status + response body.
- `login` accept timeout → exit non-zero with a clear message.

## Testing

- **Unit:** PKCE generation (verifier length/charset; challenge equals `base64url(SHA256(verifier))`);
  callback URL parsing + `state` verification (match, mismatch, `error` param); config
  read/write round-trip preserving mode 0600 and other profiles/fields.
- **Integration:** socket relay end-to-end with a local listener and a synthetic callback URL;
  `callback` fallback path when no listener is present.

## Out of scope (YAGNI)

- The token-refresh command itself — this design only **persists** the refresh token.
- Any change to the generated CLI (`cli/`).
- Non-Linux desktop scheme handlers (macOS/Windows).
